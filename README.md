数据提取阶段 \rightarrow 数据处理与清洗阶段 \rightarrow 用户确认阶段。
​第一阶段：数据准备 (现行系统侧)
​数据源 (左侧)：
​来自“G经营计划现行系统”。
​涉及多个数据库表/主数据，包括：组织Master、WBS/PJ代码Master、集团公司Master、案件WBS链接表、销售实绩信息、订单Profile、计划Profile等。
​动作：
​Dump取得：获取现行DB中迁移对象表的全部数据（为了减少对生产环境的影响，采取全件Dump的方式）。
​生成 .zip 压缩包。
​第二阶段：数据处理 (迁移团队/中间库)
​调查用DB：
​将Dump出来的数据导入到迁移团队建立的“调查用DB”中。
​这里会结合“一般会计”和“请求（请款）”的数据。
​数据筛选与生成：
​逻辑：根据请求对象（Billing）的订单号，去匹配案件数据。
​输出：生成“请求迁移对象明细”和“一般会计迁移对象”的CSV文件。
​自动检查 (ETL外)：
​利用“迁移工具”抽出需要进行WBS关联的案件详细编号。
​系统会自动检查WBS（PJ）代码是否已包含在迁移对象中。
​第三阶段：异常处理与用户作业 (右侧核心部分)
​这是流程中最关键的人工介入环节：
​判定 (NG/OK)：
​如果WBS代码存在且正确 \rightarrow 流程继续。
​如果NG（WBS代码不在迁移对象外，或存在错误） \rightarrow 生成“WBS错误CSV”。
​【用户作业】 (User Task)：
​手动操作：业务部门（事业本部）需要介入。
​任务：分析订单号和WBS的组合，识别未包含的或错误的WBS。
​WBS关联移行调书：系统或迁移团队会将“WBS关联移行调书”（Excel/CSV形式）发送给各部门。
​修正与反馈：用户在Excel中手动填写/修正正确的WBS关联信息，然后发回给处理方。# aws-quiz-site
